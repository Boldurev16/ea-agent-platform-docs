# Executive overview: зачем нужна агентная система EA

`ea-agent-platform` — учебный пример self-hosted агентной системы для управления корпоративной архитектурой. Она объединяет диалоговый интерфейс, LLM, инструменты агента, корпоративную базу знаний и контур загрузки документов.

## Управленческая проблема

В крупной компании архитектурные знания распределены между политиками, презентациями, техническими радарами, регламентами, решениями комитетов и экспертами. Даже если документы существуют, они часто не превращаются в быстрый и проверяемый ответ на вопрос руководителя или архитектора.

Агентная система решает эту проблему не “магией LLM”, а управляемым контуром:

1. пользователь задает вопрос;
2. агент определяет, нужен ли поиск по базе знаний;
3. поиск возвращает релевантные фрагменты;
4. агент формирует ответ;
5. пользователь видит источники и может проверить основание рекомендации.

## Почему это важно для C-level

| Бизнес-задача | Как помогает платформа |
|---------------|------------------------|
| Ускорить подготовку архитектурных ответов | Диалоговый доступ к корпоративной базе знаний сокращает время поиска и первичного анализа. |
| Повысить доверие к AI-ответам | Citations показывают, какие документы использованы. |
| Снизить зависимость от отдельных экспертов | Знания становятся частью управляемого контура, а не только личной памяти специалистов. |
| Поддерживать единый архитектурный язык | Один агент корпоративной архитектуры формирует ответы в общей терминологии. |
| Подготовить базу для product-grade UX | Backend уже поддерживает RAG, streaming, readiness и async ingest. |

## Что реализовано в текущем контуре

- Единый агент корпоративной архитектуры.
- RAG on-demand через инструмент `search_kb`.
- Потоковый чат `POST /chat/agent`.
- Корпоративная база знаний на основе embeddings и vector store.
- Управляемая загрузка документов через ingestion pipeline и worker queue.
- Readiness/liveness для эксплуатации.

## Что пока не является продуктовой оболочкой

Текущий UI — учебный и демонстрационный. Для полноценного бизнес-продукта нужны workspaces, загрузка документов drag-and-drop, роли пользователей, админ-консоль и коннекторы к корпоративным источникам. Эти направления выделены как Phase 2.
