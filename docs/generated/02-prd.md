# 02 — PRD текущего MVP

Документ описывает реализованный контур как продуктовую основу. Это не roadmap и не полный enterprise-продукт; это зафиксированный MVP, на котором можно строить пользовательскую оболочку и расширения.

## 1. Название

**ea-agent-platform** — платформа агента корпоративной архитектуры с RAG-контуром и управляемой загрузкой знаний.

## 2. Продуктовая гипотеза

Если архитектурные документы компании доступны агенту через управляемую базу знаний, а ответы сопровождаются источниками, то архитектурная функция может быстрее готовить консультации, снижать риск разрозненных трактовок и повышать доверие к AI-инструментам.

## 3. Пользователи

| Пользователь | Потребность |
|--------------|-------------|
| Руководитель архитектурной функции | Получить быструю и объяснимую позицию по вопросу. |
| Enterprise architect | Найти релевантные стандарты, политики и решения. |
| Оператор базы знаний | Обновлять корпус документов и видеть статус обработки. |
| Инженерная команда | Расширять agent runtime, tools и retrieval. |
| Эксплуатация | Диагностировать готовность сервиса и зависимостей. |

## 4. Реализованные пользовательские сценарии

### Сценарий A — вопрос к архитектурному агенту

1. Пользователь задает вопрос через чат или API.
2. Агент получает историю сессии.
3. LLM решает, нужен ли поиск в базе знаний.
4. При необходимости вызывается `search_kb`.
5. Пользователь получает потоковый ответ с источниками.

**Ценность:** пользователь видит не только текст ответа, но и основание, на котором он сформирован.

### Сценарий B — обновление базы знаний

1. Оператор добавляет документы в контур ingestion.
2. Система создает задание обработки.
3. Worker разбирает документы, делит на фрагменты, строит embeddings и индексирует.
4. Статус задания меняется до `done` или `failed`.

**Ценность:** пополнение знаний становится управляемым процессом, а не ручной операцией “положили файл и забыли”.

### Сценарий C — эксплуатационная диагностика

1. Support проверяет liveness.
2. Затем проверяет readiness.
3. Если readiness degraded, смотрит, какая зависимость блокирует готовность.

**Ценность:** инцидент локализуется по контуру: приложение, LLM, embeddings, vector store, storage или база данных.

## 6. Non-functional requirements

| Качество | Требование |
|----------|------------|
| Трассируемость | Ответ должен иметь источники, если использовался RAG. |
| Управляемость | Загрузка документов идет через pipeline и очередь. |
| Развертываемость | Компоненты должны работать в self-hosted контуре. |
| Наблюдаемость | Health endpoints должны разделять “жив” и “готов”. |
| Расширяемость | Tools и provider boundaries должны позволять развитие без переписывания ядра. |
| Санитизация | Документация не должна раскрывать секреты, приватные пути и внутренние промпты. |

## 7. Границы MVP

### Входит

- Один агент корпоративной архитектуры.
- `search_kb` как основной production tool.
- RAG on-demand.
- SSE chat.
- Ingestion документов.
- Readiness/liveness.
- Async worker queue.

### Не входит

- Пользовательская аутентификация.
- Workspaces и изоляция команд.
- Полный product UI.
- Drag-and-drop document management.
- Confluence, SQL, web connectors.
- MCP и агентные flows.
- Embed-widget для портала.

## 8. Риски и ограничения

| Риск | Почему важен | Комментарий |
|------|--------------|-------------|
| Нет auth/RBAC | Нельзя безопасно расширять на несколько команд | Должно быть закрыто в product shell. |
| Один основной tool | Агент ограничен поиском по KB | Следующий шаг — SQL, Confluence, web/read_doc. |
| Нет workspaces | Сложно изолировать области знаний | Ближайшая продуктовая эволюция. |
| Качество зависит от документов | Плохой corpus дает слабый retrieval | Нужны правила ведения базы знаний. |
| Нет полноценной админ-консоли | Настройки требуют инженерного участия | Требуется self-service слой. |

## 9. Acceptance criteria текущего контура

| Критерий | Как проверить |
|----------|---------------|
| Документы превращаются в indexed knowledge | Ingestion smoke |
| Поиск возвращает citations | Retrieval smoke |
| Агент вызывает `search_kb` на вопрос по базе знаний | Agent live smoke |
| SSE поток завершается событием `close` | Chat smoke |
| Readiness показывает зависимости | M7 smoke |
| Async ingest проходит `queued → done` | M7 smoke |

## 10. Словарь PRD

- **MVP** — текущий backend-ready контур.
- **Product shell** — будущая пользовательская оболочка: workspaces, upload, auth, admin UI.
- **Корпоративная база знаний** — документы и фрагменты, доступные retrieval-контуром.
- **Трассируемость** — возможность объяснить, какие источники повлияли на ответ.
