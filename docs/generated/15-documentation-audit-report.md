# 15 — Отчет о достоверности документации

Этот отчет фиксирует не строки исходного кода, а уровень доверия к материалу. Его задача — показать, какие утверждения можно считать устойчивыми, какие являются архитектурными выводами, а какие требуют проверки в конкретном окружении.

## 1. Устойчивые утверждения

| Утверждение | Статус |
|-------------|--------|
| Основной пользовательский путь — потоковый чат `POST /chat/agent`. | Подтверждено реализацией |
| Агент использует tool loop и может вызвать `search_kb`. | Подтверждено реализацией |
| RAG на основном пути вызывается on-demand, а не обязательным pre-step. | Подтверждено реализацией |
| Ответ может включать citations и trace. | Подтверждено реализацией |
| Ingestion выполняет parse, chunk, embed и index. | Подтверждено реализацией |
| Async ingestion использует job lifecycle `queued`, `running`, `done`, `failed`. | Подтверждено реализацией |
| Readiness отличается от liveness. | Подтверждено реализацией |
| Workspaces, auth/RBAC, connectors и embed-widget относятся к будущему product shell. | Подтверждено границами MVP |

## 2. Архитектурные выводы

| Вывод | Почему важен |
|-------|--------------|
| Платформа является backend-ready контуром, но не полноценным self-service продуктом. | Помогает правильно планировать Phase 2. |
| Качество ответа зависит от качества корпоративной базы знаний. | Нельзя компенсировать плохой corpus только LLM. |
| Session memory не должна рассматриваться как источник истины. | Факты должны приходить из retrieval и citations. |
| `search_kb` — первый пример управляемого инструмента агента. | Будущие tools должны сохранять тот же подход к trace и безопасности. |

## 3. Зоны, требующие проверки при внедрении

| Зона | Что проверить |
|------|---------------|
| Производительность ingestion | Время обработки больших корпусов и параллельных jobs. |
| Качество embeddings | Релевантность на реальных вопросах архитектурной функции. |
| Workspace isolation | Отсутствие смешивания областей знаний. |
| Auth/RBAC | Модель доступа для бизнес пользователей, архитекторов, архитекторов решений. |
| Observability | Достаточно ли текущих trace и health для on-call сценариев. |
| Санитизация | Нет ли утечки внутренних путей, секретов и приватных prompt-текстов. |

## 4. Легенда достоверности

| Маркер | Значение |
|--------|----------|
| Подтверждено реализацией | Поведение соответствует реализации или smoke-проверкам. |
| Архитектурный вывод | Объяснение выведено из структуры системы и должно проверяться при изменениях. |
| Требует проверки | Нужны runtime-данные, нагрузочные проверки или бизнес-решение. |
| Product gap | Возможность нужна для бизнес-пользователя, но не входит в текущий MVP. |
